## 1. 현재 결론

이번 슬라이스는 DMS 런칭 직전 사용자 검증에서 드러난 문서 사이드카, 권한 요청/승인, 알림, 댓글, AI 요약 작성, 링크 라우팅, 협업 soft lock 흐름을 정리한 closeout 입니다.

현재 완료로 볼 수 있는 범위:

- 문서 사이드카 상태/정보/공개 범위/권한/댓글 섹션 정리
- 전역 tooltip 을 browser title 속성 기준으로 정렬
- 알림 패널의 카운트/읽음/안읽음/전체 읽음/문서 대상 자동 읽음 처리
- 권한 섹션에서 소유자의 요청 처리와 grant 관리, 열람자의 내 권한 표시와 쓰기 권한 요청
- 권한 요청/승인/회수 이벤트의 SSE 기반 실시간 반영
- 권한 회수 시 열람 중 문서 즉시 차단, 권한 승인 시 열린 문서 재조회
- AI 요약 새 문서 작성 시 원본 파일 첨부 유지와 내부/외부 링크 라우팅 보정
- 댓글을 문서 metadata 저장 흐름에서 분리해 DB relation 기반 API로 전환
- 댓글 작성/삭제/복원 권한, 문서 소유자/댓글 작성자 알림, 열린 문서 댓글 실시간 갱신
- 댓글 삭제 확인 다이얼로그와 삭제 이력 표시/복원 권한 정리
- 협업 soft lock WebSocket 문서 방 이벤트 기반 반영과 편집 중 주기 잠금 갱신
- 편집 진입 시 soft lock 서버 검증, 다른 사용자 편집 중일 때 편집 차단
- 소유자/편집 권한자 어느 쪽이 먼저 편집하든 나머지 편집 가능 사용자에게 즉시 편집 차단 상태 전파
- soft lock 실시간 전달 안정화: 서버/브라우저 문서 경로 정규화, 문서 화면 직접 WebSocket 구독 요청, WebSocket 문서 방 구독 확인 후 현재 협업 스냅샷 재조회, 편집 lock 획득 후 탭 편집 상태와 화면 모드 전환 순서 고정, 2.5초 즉시 차단 양방향 브라우저 회귀 케이스 추가
- soft lock 유령 잠금 방지: 실제 편집 세션 renew 가 끊긴 lock 과 stale 편집 presence 는 같은 사용자가 앱에 접속 중이어도 만료
- 잠금 해제 요청/승인/거절 프로세스 도입
- 해제 요청 중복 생성 방지와 새로고침 후 `요청 중`/처리 다이얼로그 복원
- 해제 요청 수신 안정화: 편집 잠금 보유자 화면은 요청 수신 시 에러 페이지로 전환되지 않고 처리 다이얼로그를 유지
- 해제 요청 처리 안정화: 보유자 거절 후 처리 다이얼로그가 다시 열리지 않고, 보유자가 승인할 때 미저장 초안은 먼저 저장된 뒤 요청자에게 최신 본문과 함께 잠금이 이전
- 해제 요청 만료 안정화: 만료된 요청에 보유자가 늦게 응답해도 기존 보유자 잠금이 유지되고, 요청자 화면은 만료 시 최신 협업 스냅샷으로 버튼 상태를 다시 계산
- 잠금 세션 fencing: 같은 사용자라도 다른 탭/브라우저 세션은 별도 편집 세션으로 차단하고, 저장/메타데이터 변경은 현재 lock 세션만 통과
- 미저장 초안 승인 UX 단일화: 보유자에게 미저장 변경사항이 있으면 첫 확인창부터 `저장 후 허용`을 표시
- 런칭 브라우저 스모크 확대 자동화: 로그인, 잠긴 검색 결과, 권한 요청/승인, 승인 후 하드 새로고침, 댓글 작성/삭제/복원, 권한 거절/회수, 내부/외부 링크 라우팅, soft lock takeover 거절/승인 흐름
- 다른 사용자 저장 반영 보강: 활성 문서는 최신 본문을 자동 반영하고, 비활성으로 열려만 있는 문서는 수정 토스트를 띄우지 않도록 브라우저 회귀 추가
- 잠긴 문서 preview-only 응답의 원문 본문 노출 차단
- 첫 접속 파일 트리 안정화: 서버 부팅 중 문서 control-plane 을 선동기화하고, missing/deleted 문서가 파일 목록에 섞이지 않도록 차단. 사용자별 파일 트리 초기화 전에는 빈 목록 대신 로딩 상태를 유지
- 검증용 임시 문서 publish 안정화: Docker 기본값에서 `launch-smoke/`, `codex-lock-ui/`, `codex-lock-probe/`, `verify-access/` 문서 prefix 를 local-only 로 제외하고, 생성 직후 삭제된 미추적 markdown 경로는 Git pathspec 실패 알림이 아니라 커밋할 변경 없음으로 처리
- Docker server/dms 재빌드와 health 확인

런칭 전 남은 핵심 리스크:

- 브라우저 기준 연속 스모크 확대분은 자동화로 통과했습니다. 다만 AI 요약 새 문서 저장과 첨부 확인은 외부 모델 의존성이 있어 별도 환경 확인이 필요합니다.
- 운영 seed, 계정, 문서 root, GitLab/GitHub 배포 브랜치 상태는 최종 런칭 전 한 번 더 동결 확인이 필요합니다.
- 저장소 어댑터와 수집 채널의 고도화 항목은 런칭 핵심 협업/권한 게이트 밖의 후속 P1/P2로 남아 있습니다.

## 7. 주의사항

- 권한/알림/댓글처럼 특정 문서를 대상으로 하는 이벤트는 SSE 도메인 이벤트를 우선 사용합니다.
- 문서 방 구독, 파일 변경, publish 상태, soft lock 처럼 문서 화면과 상호작용하는 실시간 협업 상태는 WebSocket 경로를 사용합니다.
- soft lock 은 요청자 임의 탈취가 아니라 보유자 승인/거절 기반입니다.
- 해제 요청은 서버 pending state 를 기준으로 판단해야 하며, 클라이언트 메모리만 신뢰하면 안 됩니다.
- 댓글은 문서 저장 diff 대상이 아니며, 문서 metadata 배열 저장으로 되돌리면 안 됩니다.
- 삭제 계열 액션은 공용 확인 다이얼로그를 유지합니다.
- Docker 반영 없이 “완료”로 닫지 않습니다.
